# Sticker Swap Marketplace — Design

**Date:** 2026-05-09
**Status:** Approved (pending implementation plan)

## Goal

Add a public-facing marketplace on top of the existing Panini sticker tracker so that external visitors can:

- Browse the owner's missing list and doubles list with independent search per list.
- Submit swap proposals containing one or more *trades*. Each trade offers exactly one sticker the owner is missing in exchange for 1–5 stickers from the owner's doubles.
- Track and withdraw their own proposals via a unique URL — no account or login.

The owner (and a small set of admins) reviews proposals in an in-app inbox, accepts or rejects, and later marks accepted proposals as completed or cancelled. The marketplace is purely matchmaking; sticker counts are only ever changed manually in the existing admin pages.

## Non-goals (v1)

- Counter-proposals. Owner rejects with a note instead.
- Automatic expiration of pending proposals.
- Threaded chat — only one optional note per side.
- Push or email notifications — in-app badge only.
- Automatic sticker count updates from proposal completion. Counts are always edited manually in existing pages.
- Proposer accounts or login.
- Partial accepts within a proposal — accept/reject the whole bundle.
- Multi-quantity *offered* (offered side is always exactly one sticker per trade).
- Image uploads or sticker condition tracking.

## Architecture

**One repo, two Vite entry points** — admin and marketplace are different React trees in the same project, sharing data and utilities.

| Entry | URL | Routes |
|---|---|---|
| `index.html` (admin) | `/`, `/team/:code`, `/missing`, `/doubles`, `/players`, **`/inbox`** | Existing app + new Inbox tab |
| `market.html` (public) | `/market`, `/market/new`, `/market/p/:id` | New marketplace |

One Vercel project. `vercel.json` rewrites split traffic by path:

```json
{
  "rewrites": [
    { "source": "/market/:path*", "destination": "/market.html" },
    { "source": "/market", "destination": "/market.html" },
    { "source": "/((?!market|assets|favicon).*)", "destination": "/index.html" }
  ]
}
```

Vite multi-entry config:

```ts
build: {
  rollupOptions: {
    input: {
      main: resolve(__dirname, 'index.html'),
      market: resolve(__dirname, 'market.html'),
    },
  },
},
```

The marketplace tree never imports the admin TabBar; visitors see no admin navigation.

## Data model

### `stickers/{code}` (existing, unchanged)

```ts
{ count: number; name: string | null; updatedAt: Timestamp }
```

### `admins/{uid}` (new)

One document per allowed admin. UID is the Firebase Auth UID.

```ts
{ name: string; addedAt: Timestamp }
```

### `proposals/{id}` (new)

**Doc ID is a 12-character nanoid generated by the client at creation.** This same ID is the proposer's tracking URL slug — `/market/p/<id>` maps to `proposals/<id>`. Brute-forcing the ID space is infeasible, so knowing the URL is the gate that lets the proposer self-withdraw without auth.

```ts
{
  status: 'pending' | 'accepted' | 'rejected' | 'withdrawn' | 'completed' | 'cancelled'
  trades: Array<{
    offered: string                            // sticker code from owner's Missing
    requested: Array<{ code: string; qty: number }>  // 1+ entries; sum(qty) ∈ [1, 5]
  }>
  proposer: { name: string; contact: string }
  proposerNote: string | null
  ownerNote: string | null                     // null until owner adds one
  createdAt: Timestamp
  decidedAt: Timestamp | null                  // set on accept/reject
  closedAt: Timestamp | null                   // set on any terminal state
  decidedBy: string | null                     // admin uid that decided
}
```

**Status semantics:**

| Status | Set by | Reservation active? | Terminal? |
|---|---|---|---|
| `pending` | proposer (create) | no | no |
| `accepted` | admin | **yes** | no |
| `rejected` | admin | no | yes |
| `withdrawn` | proposer (self) | no | yes |
| `completed` | admin (on accepted) | no | yes |
| `cancelled` | admin (on accepted) | no | yes |

`completed` vs `cancelled` are both terminal post-accept states with identical reservation effects. They differ only as a record of intent in the history view. `decidedAt` is set when the admin first accepts/rejects. `closedAt` is set on any transition into a terminal state.

### Reservation logic — derived, not stored

Reservation aggregates are computed in memory from the live proposal subscription, never denormalized into sticker docs:

```
incomingReserved[code] = count of accepted proposals where any trade.offered == code
outgoingReserved[code] = sum of qty across all accepted proposals' trade.requested[*] for that code
```

Lists are bounded (~960 stickers, expected < 50 active proposals). The marketplace and admin reservation badges both read these aggregates.

**Marketplace display rules:**

- **Missing list** shows every code with `stickers[code].count === 0`. If `incomingReserved[code] >= 1`, render with an "incoming" badge but still listed.
- **Doubles list** shows every code with `stickers[code].count >= 2`. Available spare = `(count - 1) - outgoingReserved[code]`. If 0 available, render with an "all reserved" badge but keep visible.

**Proposal form rules** (validated client-side, structurally checked by Firestore rules):

- Each trade: `offered` must be a code currently in Missing **and** `incomingReserved[code] === 0` (no double-promising the same incoming sticker).
- Each `requested[i].qty` ≤ available spares for that code at submission time.
- Per-trade: `sum(requested.qty) ∈ [1, 5]`.
- Proposal-level: `trades.length ≥ 1`; offered codes unique across all trades within the proposal.

## Public marketplace UI

### `/market` — landing/browse

Header with a short "About" blurb and the owner's contact handle. Two browseable lists, each with its own independent search bar:

- **"What I need"** (Missing) — grouped by team, alphabetical. Shows code, player/album name, "incoming" badge when `incomingReserved >= 1`.
- **"What I'm giving"** (Doubles) — grouped by team. Shows code, name, available count, "all reserved" badge when 0 available.

Each search bar has a small **Name / Code** segment toggle (default Name).

- **Name mode:** plain substring match against player + team name using existing `normalizeForSearch` (accent/punctuation insensitive).
- **Code mode:** matches against sticker code. Auto-dash kicks in *only in this mode* — typing exactly a known team code (e.g., `POR`) auto-completes the input to `POR-` so the next digit filters to `POR-N`. Backspace through the dash like normal text.

Floating "Send a swap proposal" button → `/market/new`.

### `/market/new` — trade builder

Single form holding a list of trade rows (start with 1; "Add another trade" button appends rows).

Each trade row:

- **Offered:** single picker that searches Missing-where-not-already-reserved-incoming. Click to pick. Exactly one offered sticker per trade.
- **Requested:** 1–5 picks from Doubles. Each pick is `{code, qty}` with a qty stepper bounded by the code's available spares. Running total: `2 / 5 requested`.
- **Per-trade ratio badge:** `1 : 3` — green when valid, red and submission disabled when `sum(requested.qty)` is 0 or > 5.

Below the trades:

- **Add another trade** button.
- **Your details** — name (required), contact (required, free text — email/phone/Instagram/etc.), optional note.
- **Submit** — disabled until every trade row is valid and proposer fields are non-empty.

On submit:

1. Re-validate fully on the client.
2. Generate a 12-char nanoid for the doc ID.
3. Write `proposals/<id>` with `status: 'pending'`.
4. Show a success page with the tracking URL, a copy-to-clipboard button, and "save this link to check status" guidance.

### `/market/p/:id` — proposer's tracking page

Reads the single proposal by doc ID.

Shows:

- Status banner with relevant timestamps (Pending / Accepted / Rejected / Withdrawn / Completed / Cancelled).
- All trades laid out read-only.
- Proposer note + owner note (if set).
- After acceptance: owner's contact info appears here so the proposer can reach the owner.
- "Cancel proposal" button — visible only when `status === 'pending'`. Flips to `withdrawn` and sets `closedAt`.

Unknown ID → "Proposal not found" page with a link back to `/market`.

## Admin inbox UI

New `/inbox` route in the admin app, plus a new "Inbox" tab in the existing TabBar with a red-dot badge showing pending count. The tab is only rendered when the user is signed in as an admin.

### `/inbox` page

Top filter chips: **Pending** (default) · **Accepted** · **History** (rejected / withdrawn / completed / cancelled).

Each list is a stack of **proposal cards**, sorted newest-first. Each card shows:

- Status pill, age, proposer name + contact.
- All trades laid out compactly: `POR-5 → BRA-1 ×2, BRA-2` with player/team names visible.
- Proposer note (if any).
- Action row by status:
  - **Pending** → `Accept` / `Reject` buttons; tapping either opens a confirm sheet with optional ownerNote field.
  - **Accepted** → `Mark completed` and `Cancel swap` buttons + an inline summary of what's currently reserved by this proposal (e.g., "reserves: 1× POR-5 incoming · 2× BRA-1, 1× BRA-2 outgoing").
  - **Terminal states** → recorded ownerNote (if any), no actions.

### Action handlers

- **Accept:** write `{ status: 'accepted', decidedAt: serverTimestamp(), decidedBy: uid, ownerNote }`. Reservation derivation picks it up automatically.
- **Reject:** write `{ status: 'rejected', decidedAt: serverTimestamp(), closedAt: serverTimestamp(), decidedBy: uid, ownerNote }`.
- **Mark completed:** write `{ status: 'completed', closedAt: serverTimestamp() }`. **No automatic sticker count changes.** Owner updates counts manually in existing pages whenever the real-life exchange happens.
- **Cancel swap:** write `{ status: 'cancelled', closedAt: serverTimestamp() }`. Same: no count changes; just clears the reservation badge and records that the swap fell through.

### Reservation badges in existing admin pages

Existing Missing and Doubles pages get the same reservation badges as the marketplace, computed from the same in-memory aggregate. So while scrolling Doubles in admin, you see "BRA-1 — 3 spare (1 reserved out)" matching what the marketplace shows.

## Auth & security

### Sign-in flow

- Google sign-in via Firebase Auth `signInWithPopup`.
- After sign-in, app reads `admins/{auth.currentUser.uid}`. If missing, the user gets a "Not authorized" page that surfaces their UID so an existing admin can add them.
- A small `<AuthGate>` component wraps `/inbox` and the existing pages' write-action handlers (sticker increment/decrement). Read views remain accessible without sign-in to preserve current public-browsing UX.
- To add a second admin: any current admin creates a new doc in `admins/{newUid}` with the new admin's UID.

### Firestore security rules

```
rules_version = '2';
service cloud.firestore {
  match /databases/{db}/documents {
    function isAdmin() {
      return request.auth != null
        && exists(/databases/$(db)/documents/admins/$(request.auth.uid));
    }

    match /admins/{uid} {
      allow read: if request.auth != null && request.auth.uid == uid;
      allow write: if isAdmin();
    }

    match /stickers/{code} {
      allow read: if true;
      allow write: if isAdmin();
    }

    match /proposals/{id} {
      allow read: if true;

      // Public create — pending only, with structural validation
      allow create: if
        request.resource.data.status == 'pending'
        && request.resource.data.createdAt == request.time
        && request.resource.data.decidedAt == null
        && request.resource.data.closedAt == null
        && request.resource.data.decidedBy == null
        && request.resource.data.ownerNote == null
        && request.resource.data.proposer.name is string
        && request.resource.data.proposer.name.size() > 0
        && request.resource.data.proposer.contact is string
        && request.resource.data.proposer.contact.size() > 0
        && request.resource.data.trades.size() >= 1
        && request.resource.data.trades.size() <= 20;

      // Proposer self-withdraw: pending → withdrawn, only status + closedAt change
      allow update: if
        resource.data.status == 'pending'
        && request.resource.data.status == 'withdrawn'
        && request.resource.data.diff(resource.data)
             .affectedKeys().hasOnly(['status', 'closedAt']);

      // Admin updates (accept/reject/complete/cancel/note)
      allow update: if isAdmin();

      // No public deletes; admin-only cleanup if ever needed
      allow delete: if isAdmin();
    }
  }
}
```

Per-trade ratio validation and offered-uniqueness are enforced client-side. Firestore rules can't elegantly walk nested arrays; structural rules above prevent the worst tampering and the admin sees and can reject malformed proposals at decision time.

### Existing admin write UX without auth

The increment/decrement controls on `StickerSheet` show a "Sign in to edit" prompt and disable while no auth is present. Read flows are unchanged.

## State management

`src/lib/state.ts` (existing zustand store) extended with:

- `proposals: Map<id, Proposal>`
- `subscribeProposals(filter)` — admin app subscribes to all proposals; market app subscribes only to `status === 'accepted'` for reservation derivation, plus a single proposal when viewing `/market/p/:id`.
- `incomingReserved: Map<code, number>`, `outgoingReserved: Map<code, number>` — derived selectors recomputed when proposals change.

The existing `useStickersMap`, `useTotals`, `useTeamProgress` hooks compose with reservation maps in marketplace and inbox views.

## Deployment & ops

- **Firebase console one-time:** enable Google sign-in provider; add production domain to authorized domains. No new env vars beyond existing `VITE_FIREBASE_*`.
- **Rules deploy:** `firebase deploy --only firestore:rules` after `npm i -g firebase-tools` and `firebase login`.
- **Vercel:** existing project, no plan upgrade needed. Hobby tier supports custom domains and multi-entry Vite builds out of the box.
- **First admin bootstrap:** sign in with Google to land on the "Not authorized" page, copy the UID it shows, manually create `admins/{thatUid}` in the Firebase console.
